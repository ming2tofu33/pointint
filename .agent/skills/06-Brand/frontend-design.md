# Pointint Frontend Design

## Context

### 기술 스택

- **Framework:** Next.js (App Router)
- **Hosting:** Vercel
- **Styling:** Tailwind CSS
- **Components:** shadcn/ui 기반
- **Backend:** FastAPI + Supabase

### 브랜드 디자인 원칙

- 여백이 넉넉하다
- 색은 분위기를 만든다 (기능 구분이 아님)
- 깔끔하되 차갑지 않다
- 커서 결과물이 비주얼의 중심이다
- 갤러리처럼 보이면 안 된다
- 기술 도구처럼 보이면 안 된다

### 핵심 흐름

```
이미지 업로드 → 편집 → Hotspot 지정 → 시뮬레이션 → 다운로드
```

### 타깃 기준

일반 사무직이 막힘 없이 완료할 수 있으면, 나머지 그룹은 더 쉽게 쓸 수 있다.

## Workflow

### 1. 시스템 스킬 체이닝

`frontend-design` 시스템 스킬 호출 → Pointint 브랜드/스택 맥락 적용

### 2. 구현 전 체크리스트

- [ ] 이 페이지/컴포넌트의 역할이 핵심 흐름에서 어디인가?
- [ ] Pointint 톤에 맞는가? (여백, 짧은 카피, 느낌 전달)
- [ ] 일반 사무직 기준으로 직관적인가?
- [ ] 불필요한 네비게이션/UI 요소가 없는가?
- [ ] 데스크톱 우선인가? (커서 = 데스크톱 제품)

### 3. 컴포넌트 설계 원칙

**Pointint UI 원칙:**
- CTA는 명확하되 강요하지 않는다
- 단계 전환은 자연스럽게 (끊김 없이)
- 로딩/대기는 시각적 피드백으로 채운다
- 결과물(커서)을 항상 중심에 둔다
- 에러 메시지도 Pointint 톤 (친절하되 간결)

### 4. 페이지별 방향

| 페이지 | 핵심 | 톤 |
|---|---|---|
| 랜딩 | 업로드 CTA + 결과물 예시 | 감성, 여백 |
| 에디터 | 제작 워크플로우 중심 | 도구적이되 차갑지 않게 |
| 시뮬레이션 | 실사용 미리보기 | 기대감, 확인의 순간 |
| 다운로드 | 완성 만족감 + 적용 가이드 | 완성의 뿌듯함 |
| 마이페이지 | 프로젝트 목록 + 설정 | 깔끔, 효율 |
| 마켓플레이스 | 커서 테마 탐색 + 판매 | 갤러리 아님, 큐레이션 |

### 5. 변경 반영

1. 기능 문서: `point/03-Features/` 관련 문서 참조/업데이트
2. 디자인: `point/11-Design/` 관련 문서 참조/업데이트
3. 구현 코드: 프로젝트 소스 디렉토리

## Constraints

- 비주얼 톤앤매너(색상 팔레트) 미확정 시 → 중립 팔레트로 진행, 후 교체 용이하게
- 모바일은 후순위 — 데스크톱 우선 설계
- Phase 1 Wave 1은 프로젝트 셋업부터 시작
