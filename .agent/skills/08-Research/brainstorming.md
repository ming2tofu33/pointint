# Pointint Brainstorming

## Context

### 제품 핵심

- 이미지를 Windows 커서로 만드는 가장 쉬운 제작 워크플로우
- `Your Point, Your Tint.`
- 브랜드는 넓게, 첫 제품은 좁게

### 의사결정 원칙

1. **일반 사무직 기준으로 쉬운가?** — 이 그룹이 막힘 없으면 나머지도 OK
2. **제작 워크플로우를 강화하는가?** — 다운로드/갤러리가 아닌 제작
3. **MVP 범위 안인가?** — Phase 1은 업로드 기반 커서 제작 검증
4. **Pointint 톤에 맞는가?** — 기능 나열 ✗, 느낌 전달 ✓

---

## Workflow

### 1. 시스템 스킬 체이닝

`brainstorming` 시스템 스킬 호출 → Pointint 컨텍스트 주입

### 2. 브레인스토밍 프레임

**Step 1: 의도 파악**
- 이 기능/변경의 본질적 목적은?
- 어떤 타깃 세그먼트에게 가장 중요한가?
- 핵심 흐름 중 어디에 위치하는가?

**Step 2: 선택지 탐색**
- 최소 2-3가지 접근 방식 제시
- 각 방식의 트레이드오프 분석
- Pointint 원칙과의 정합성 평가

**Step 3: 판단 기준**
- 구현 난이도 vs 사용자 가치
- MVP 범위 적합성
- 확장성 (모니테리어 방향과 충돌 안 하는가?)

**Step 4: 합의 & 문서화**
- 선택된 방향 요약
- 필요 시 `point/06-Implementation/plans/` 계획 문서 생성

### 3. Pointint Inbox 연동

좋은 아이디어가 나왔지만 지금 당장 실행하지 않을 때:
- `point/00-Inbox/` 에 Quick Capture로 남긴다
- 이후 정리하여 적절한 폴더로 승격

### 4. 출력 형식

```markdown
## 브레인스토밍: [주제]

### 배경
[왜 이 논의가 필요한가]

### 선택지
1. **[방식 A]** — [설명] / 장점: / 단점:
2. **[방식 B]** — [설명] / 장점: / 단점:
3. **[방식 C]** — [설명] / 장점: / 단점:

### 추천
[추천 방향 + 근거]

### 다음 단계
[합의 후 어떻게 진행할지]
```

## Constraints

- 구현 전에 반드시 브레인스토밍 — 코드부터 짜지 않는다
- CLAUDE.md의 Planning Rules 준수 (큰 작업은 plan 문서 먼저)
- Inbox 메모는 날것 OK — 정리는 나중에
